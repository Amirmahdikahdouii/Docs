## What is a TLS Handshake?

TLS or **Transport Security Protocol** is a secure protocol that designed to facilitate privacy and data security for communications over internet.
TLS could be used for encrypt data between servers and web applications, it also could be useful in email communications, messaging and [Voice over IP(VoIP)](https://www.cloudflare.com/learning/video/what-is-voip/).

> [!NOTE]
> TLS protocol has different versions but the one that is most used nowadays is Version 1.3

## TLS vs SSL

### What is SSL?

Secure Socket Layer or **SSL** is an encryption based internet security protocol, developed by Netscape.

> [!IMPORTANT]
> SSL is the predecessor to the modern TLS encryption used today.
> A website that implements **SSL/TLS**  has **HTTPS** in it's URL instead of **HTTP**

### What is the differences between TLS and SSL?

TLS evolved from SSL, TLS version 1.0 actually began development as SSL version 3.1, but the name of the protocol was changed before publication in order to indicate that it is no longer associated with Netscape.

## TLS vs HTTPS

HTTPS is an implementation of TLS encryption on top of the `HTTP` protocol, so any website that uses HTTPS is therefore employing TLS encryption.

> [!NOTE]
> Businesses and web applications should use HTTPS instead of HTTP because of it is more secure and reliable.
> Some browsers like `Google Chrome` block or warn websites that use **HTTP** instead of **HTTPS**

## What does TLS do?

In fact, TLS in separated in 3 main sections:

1. **Encryption**: Hide the data being transferred from third parties
2. **Authentication**: ensures that the parties exchanging information are who they claim to be
3. **Integrity**: verifies that the data has not been forged or tampered with

A TLS connection is initiated using a sequence known as the **TLS Handshake**.
During the TLS Handshake, the client and server:

- Specify which version of TLS they will use.
- Decide on which cipher suites they will use
- Authenticate the identity of the server using the server TLS's certificate
- Generate session keys for encrypting messages between them after the handshake is complete

> [!IMPORTANT]
> The TLS Handshake create a cipher suite for each communication session. The cipher suite is a set of algorithms that specifies details such as which shared [encryption keys](https://www.cloudflare.com/learning/ssl/what-is-a-cryptographic-key/), or [session keys](https://www.cloudflare.com/learning/ssl/what-is-a-session-key/) will be used for that particular session.
> TLS is able to set the matching session keys over an unencrypted channel thanks to a technology known as [public key cryptography](https://www.cloudflare.com/learning/ssl/how-does-public-key-encryption-work/)

The TLS handshake also check the server identity for proving to client. This is done using **public keys**. The server public key is a part of its TLS certificate.

Once data is encrypted and authenticated, it is then signed with a message authentication code (MAC). The recipient can then verify the MAC to ensure the integrity of the data.

# **How TLS 1.3 Handshake Works: A Detailed Explanation**

Transport Layer Security (TLS) is a cryptographic protocol designed to provide secure communication over a computer network. TLS 1.3, the latest version of the protocol, introduces significant improvements over its predecessors, including faster handshakes, enhanced security, and simplified cryptographic negotiation. In this article, we will explore the TLS 1.3 handshake process in detail, breaking down each step, the data exchanged, and how the client verifies the server's identity.

---

## **1. Overview of the TLS 1.3 Handshake**

The TLS 1.3 handshake is the process by which a client and server establish a secure connection. Unlike earlier versions of TLS, TLS 1.3 reduces the number of round trips required for the handshake, making it faster and more efficient. The handshake involves the following key steps:

1. **Client Hello**
2. **Server Hello**
3. **Key Exchange**
4. **Server Certificate and Authentication**
5. **Client Certificate (Optional)**
6. **Finished Messages**

Let’s dive into each step in detail.

## **2. Step-by-Step Breakdown of the TLS 1.3 Handshake**

### **Step 1: Client Hello**
The handshake begins with the client sending a **Client Hello** message to the server. This message contains the following information:

#### **Data Sent:**
- **Supported TLS Version**: The client indicates that it supports TLS 1.3.
- **Random Bytes**: A 32-byte random value generated by the client, which will be used later in key generation.
- **Cipher Suites**: A list of cryptographic algorithms supported by the client, ordered by preference. TLS 1.3 simplifies this list by removing insecure algorithms.
- **Key Share**: The client includes one or more **key shares**, which are public keys for key exchange algorithms (e.g., Elliptic Curve Diffie-Hellman (ECDH)).
- **Session ID (Optional)**: If the client is attempting to resume a previous session, it includes the session ID.
- **Extensions**: Additional information, such as supported application-layer protocols (e.g., HTTP/2) or server name indication (SNI).

#### **Purpose:**
- The **Client Hello** initiates the handshake and informs the server of the client’s capabilities.
- The **key share** allows the server to start generating shared secrets immediately, reducing the number of round trips.

---

### **Step 2: Server Hello**
Upon receiving the **Client Hello**, the server responds with a **Server Hello** message. This message contains the following information:

#### **Data Sent:**
- **Selected TLS Version**: The server confirms that it will use TLS 1.3.
- **Random Bytes**: A 32-byte random value generated by the server, which will be used in key generation.
- **Selected Cipher Suite**: The server chooses a cipher suite from the list provided by the client.
- **Key Share**: The server includes its own **key share**, which is a public key for the chosen key exchange algorithm.
- **Session ID (Optional)**: If resuming a session, the server includes the session ID.
- **Extensions**: Additional information, such as the selected application-layer protocol.

#### **Purpose:**
- The **Server Hello** confirms the cryptographic parameters for the connection.
- The server’s **key share** allows both parties to compute a shared secret, which will be used to derive encryption keys.

---

### **Step 3: Key Exchange**
After the **Server Hello**, the server and client perform a key exchange to establish a shared secret. This step involves the following:

#### **Data Sent:**
- **Shared Secret Calculation**: Using the client’s and server’s key shares, both parties compute a shared secret using the chosen key exchange algorithm (e.g., ECDH).
- **Handshake Secrets**: The shared secret is used to derive **handshake secrets**, which are temporary keys used to encrypt the rest of the handshake.

#### **Purpose:**
- The key exchange ensures that both parties have a shared secret without transmitting it over the network, providing forward secrecy.
- The **handshake secrets** protect the remaining handshake messages from eavesdropping.

### **Step 4: Server Certificate and Authentication**
The server sends its digital certificate to the client to prove its identity. This step includes the following:

#### **Data Sent:**
- **Server Certificate**: The server’s certificate, which contains:
  - The server’s public key.
  - The server’s identity (e.g., domain name).
  - A digital signature created by a Certificate Authority (CA).
- **Certificate Verify (Optional)**: If the server uses a certificate that requires proof of possession (e.g., an ECDSA certificate), it sends a **Certificate Verify** message. This message contains a signature created using the server’s private key, proving that the server owns the public key in the certificate.

#### **Purpose:**
- The **server certificate** allows the client to verify the server’s identity.
- The **Certificate Verify** message ensures that the server possesses the private key corresponding to the public key in the certificate.

---

### **Step 5: Client Certificate (Optional)**
In some cases, the server may request the client to authenticate itself using a certificate. This step is optional and depends on the server’s configuration.

#### **Data Sent:**
- **Client Certificate**: The client’s certificate, which contains:
  - The client’s public key.
  - The client’s identity.
  - A digital signature created by a CA.
- **Certificate Verify (Optional)**: If the client uses a certificate that requires proof of possession, it sends a **Certificate Verify** message.

#### **Purpose:**
- The **client certificate** allows the server to verify the client’s identity.
- This step is typically used in mutual TLS (mTLS) scenarios, where both parties authenticate each other.

---

### **Step 6: Finished Messages**
The handshake concludes with both parties sending **Finished** messages to verify the integrity of the handshake.

#### **Data Sent:**
- **Finished Message**: A message containing a hash of all previous handshake messages, encrypted using the handshake keys.
- **Application Keys**: Both parties derive **application keys** from the shared secret, which will be used to encrypt application data.

#### **Purpose:**
- The **Finished** messages ensure that the handshake was not tampered with.
- The **application keys** enable secure communication for the rest of the session.

## **3. How the Client Verifies the Server’s Identity**

The client verifies the server’s identity during the **Server Certificate and Authentication** step. Here’s how this process works:

### **Step 1: Retrieve the Server’s Certificate**
- The server sends its certificate to the client. This certificate contains:
  - The server’s public key.
  - The server’s identity (e.g., domain name).
  - A digital signature created by a CA.

### **Step 2: Verify the Certificate Chain**
- The client checks if the server’s certificate is signed by a trusted CA.
- If the server’s certificate is signed by an **intermediate CA**, the client must validate the entire certificate chain:
  1. The server’s certificate is signed by the intermediate CA.
  2. The intermediate CA’s certificate is signed by the root CA.
- The client uses the root CA’s public key (from its trusted certificate store) to verify the intermediate CA’s certificate, and then uses the intermediate CA’s public key to verify the server’s certificate.

### **Step 3: Decrypt the Digital Signature**
- The server’s certificate contains a digital signature, which is the hash of the certificate’s contents encrypted with the CA’s private key.
- The client uses the CA’s public key to decrypt the signature, producing the original hash value.

### **Step 4: Compute the Hash of the Certificate**
- The client independently computes the hash of the server’s certificate (excluding the signature) using the same hashing algorithm (e.g., SHA-256) that the CA used.

### **Step 5: Compare the Hashes**
- The client compares the decrypted hash (from Step 3) with the computed hash (from Step 4).
- If the hashes match, it proves:
  1. The certificate has not been tampered with.
  2. The certificate was indeed signed by the trusted CA.

### **Step 6: Validate the Server’s Identity**
- The client checks that the domain name in the certificate matches the server’s actual domain name (e.g., the URL the client is connecting to).
- It also checks the certificate’s validity period to ensure it has not expired.

## **4. Benefits of TLS 1.3**

TLS 1.3 offers several advantages over previous versions:
- **Faster Handshake**: The handshake is completed in one round trip, reducing latency.
- **Enhanced Security**: Insecure algorithms and features (e.g., RSA key exchange, static DH) have been removed.
- **Forward Secrecy**: Ephemeral key exchange ensures that session keys are not compromised even if the server’s private key is leaked.
